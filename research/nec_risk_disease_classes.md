# High-NEC-risk disease classes in the dismech knowledge base

_Deliverable for acceptance-criterion item 3 of issue #3889
("Identify high-NEC-risk disease classes"). Generated by
`scripts/nec_risk_audit.py` over the 1,166 current **non-history**
`kb/disorders/*.yaml` entries (the scan excludes `*.history.yaml`, so this is
fewer than the ~1,314 total `.yaml` files on disk), with a curated
interpretation layer added by hand. Counts below are a point-in-time snapshot;
regenerate the raw scan with
`uv run python scripts/nec_risk_audit.py --markdown`._

## Why this exists

**Named Entity Confusion (NEC)** is the deep-research failure mode tracked in
#3889: a DR tool (Falcon, OpenScientist, …) resolves the queried disease name
to a *different* disease entity and produces a report that is *coherent but
wrong*. Citations may be real and snippets may validate against their
abstracts — but they describe the wrong disease. The existing dismech
anti-hallucination stack (snippet-in-abstract validation, PMID existence,
ontology term validation) **cannot** catch NEC, because a wrong-entity report
validates correctly against its own real wrong-disease sources. The only
reliable catch is semantic: cross-checking the report's primary gene/identity
against the authoritative MONDO mapping.

Two confirmed NEC cases motivated the issue:

| Case | Queried | DR reported | Trigger |
|---|---|---|---|
| PR #3874 | Lichtenstein-Knorr syndrome (SCAR19 / SLC9A1) | SCAR20 / SNX14 | synonym aliasing + numbered-series drift |
| PR #3835 | Temtamy syndrome (C12orf57) | Temtamy preaxial brachydactyly (CHSY1) | eponymic collision |

The per-report gene-identity check is implemented in
`src/dismech/preflight_dr.py` / `just preflight-dr` (PR #3902). **This document
is the complementary corpus-level artifact**: it enumerates which dismech
entries fall into each structural NEC-risk class, so a curator can apply
*extra* scrutiny to a DR report before trusting it, and so the priority
dashboard can eventually carry an NEC-risk flag (the dashboard-annotation half
of item 3 is left for maintainer steer — see "Next steps").

The flagger is a *risk-class* detector, **not** an assertion that any specific
entry is confused. It is intentionally high-recall; the eponym list below is
the hand-verified high-precision subset.

## 2. Synonym aliasing (104 entries)

The Lichtenstein-Knorr pattern: a historical synonym maps to a different
authoritative entity. These entries carry a synonym whose eponym differs from
the primary name — a DR tool keying off the synonym can drift. High-value
examples (full list in the raw scan):

## 3. Numbered / lettered series (81 entries)

Series whose numbering has shifted across the literature (SCAR1-21, CMT types,
CDG subtypes, atelosteogenesis I/II/III, osteopetrosis types, …). The
Lichtenstein-Knorr/SCAR20 case lives here. The synonym field frequently
exposes the drift directly, e.g. `Autosomal_Recessive_Ataxia_Beauce_Type`
(MONDO:0012549) carries both `SCAR8` and `type 1`, and `CALFAN_Syndrome`
carries `SCAR21`/`type 21`. Any DR query against a member of a numbered series
should be preflight-checked against the MONDO ID.

## 4. Acronym ambiguity (278 entries)

Short all-caps synonyms that are easy to mis-expand. The sharpest example for
dismech: `Acute_Intermittent_Porphyria` (MONDO:0008294) carries the synonym
**`AIP`**, which also names the *AIP gene* and AIP-related pituitary adenoma
predisposition (see #3691) — an acronym collision between a porphyria and a
pituitary-adenoma predisposition. Acronyms should never be the sole anchor a
DR query resolves on.

## Recommended use

1. **Before trusting a DR report**, run the per-report identity check:
   `just preflight-dr research/<report>.md MONDO:XXXXXXX` (#3902). If the
   queried disease appears in section 1, 2, or 3 above, treat a `WARN`/`SKIP`
   as seriously as a `FAIL` and verify the gene by hand.
2. **When curating any disease in a collision/series/aliasing class**, record
   the authoritative gene + OMIM in the entry so future DR runs have a
   ground-truth to check against.
3. **Numbered series + eponym collisions** are where reclassification drift
   concentrates — prefer the gene-axis `<GENE>-related …` naming convention
   (as #3310 does for gametogenic-failure spectra) to make the intended entity
   unambiguous.
